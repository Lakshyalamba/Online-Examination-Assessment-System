# Shared Component Patterns Inspiration Prompt

## Component Purpose

Define the reusable interaction language that unifies the product.

## Key Visual Hierarchy

- buttons and button groups
- input and select states
- tabs and badges
- alerts and toasts
- empty and loading states

## Target Aesthetic

Cohesive, token-driven, and restrained.

## Important Sections And Components

- primary, secondary, ghost, and destructive buttons
- text inputs, selects, textareas
- status badges
- alerts and toast examples
- skeletons and empty-state cards

## Interaction Feel

Consistent and accessible, with obvious active, focus, disabled, and error states.

## Layout Constraints

- arrange components in a system board format
- keep labels and states readable

## Consistency Rules From `design.md`

- one spacing scale
- one border-radius family
- one semantic color mapping for state
